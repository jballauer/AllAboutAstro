# K&E stub placeholder

A violet, dashed-border disclosure box (`.ke-ai-stub`, defined once in
`BaseLayout.astro`) for a cataloged K&E slide-rule model Jay has never
owned, handled, or researched — used instead of inventing plausible-
sounding first-person history that would misrepresent the book's actual
epistemic basis (the K&E book is written throughout in Jay's first-
person collector voice: he speaks with authority only about rules he's
personally handled, dated, priced, photographed).

**Scope: K&E/slide-rule content only** (`src/content/sliderules/**`).
If you're working on an Astro Gallery page instead, stop and use
`gallery-ai-stub` — same box, different fields, different rules,
different footer links.

## The authorship test — not a factual-accuracy test

**Every model name/number that appears in the source material gets its
own heading, even a bare mention with zero prose — never merge one
model's mention into another's section, never bullet-list several bare
models together (even under a shared family heading).** Given that:

1. **If Jay's own gathered source material has real facts** for the
   model (a catalog scan he provided, a renumbering cross-reference,
   anything he'd read/verify himself) — write plain sourced prose, no
   stub box. That's accurately-sourced content in the book's own voice,
   not AI research.
2. **If facts exist only because an AI assistant went and found them**
   — via web research, a collector site like Clark McCoy's K&E catalog
   pages, osgalleries.org, or similar — **it still needs the stub box**,
   even if the facts are real, accurate, and well-cited. The test is
   *authorship/voice*, not whether the facts turned out to be true.
   (This was a real correction mid-project: two Chapter 6 entries with
   genuine third-party-sourced specs were first written as plain prose
   with citations, reasoning "real facts exist elsewhere" — Jay
   corrected that the exemption in point 1 only covers facts *Jay*
   gathered, not facts *Claude* researched, regardless of accuracy.)
3. **Only reach for the stub when a model genuinely has no real
   supporting facts anywhere** in the gathered source material — and
   even then, populate whatever real facts DO exist (dates, family-wide
   facts Jay already stated elsewhere, real specs found via research)
   into the stub's fields rather than leaving it generic speculation.

**Check real sources before writing a stub at all** — Clark McCoy's K&E
catalog site (`mccoys-kecatalogs.com`) and osgalleries.org both document
real out-of-catalog and custom K&E rules with photos and specs. A bare
model name is not necessarily undocumented anywhere; check before
treating it as pure speculation.

## Structure — three variants depending on what's available

**No photo found** (most common for genuinely obscure models):

```html
<div class="ke-ai-stub">
<span class="ke-ai-stub-badge">🤖 AI-drafted &middot; unverified</span>

<dl class="ke-ai-stub-facts">
<dt>What it is</dt>
<dd>...</dd>
<dt>What it did</dt>
<dd>...</dd>
<dt>Approx. introduced</dt>
<dd>...</dd>
</dl>

<p class="ke-ai-stub-desc">This summary was generated by an AI assistant from general references, not from Jay's own collection, handling, or writing. Treat every detail above as a starting point for research, not settled fact.</p>

<div class="ke-ai-stub-media">catalog image not yet sourced</div>

<div class="ke-ai-stub-footer">
<span class="tag">Verify further:</span> <a href="https://www.oughtred.org/">The Oughtred Society</a> &middot; <a href="https://www.sliderulemuseum.com/">International Slide Rule Museum (ISRM)</a>
</div>
</div>
```

**A real photo was found** (e.g. on McCoy's site) — swap
`.ke-ai-stub-media` for `.ke-ai-stub-photo`, wire `data-hires` for the
lightbox, credit the source in the caption, and add that source to the
footer alongside the standard links:

```html
<div class="ke-ai-stub-photo" data-hires="/images/ke-rules/<hires-file>.jpg">

![<descriptive alt text>](./_images/<local-file>.jpg)

<p class="ke-figure-caption">Photo courtesy Clark McCoy, mccoys-kecatalogs.com.</p>

</div>

<div class="ke-ai-stub-footer">
<span class="tag">Verify further:</span> <a href="https://www.mccoys-kecatalogs.com/...">Clark McCoy's K&E Catalog Site</a> &middot; <a href="https://www.oughtred.org/">The Oughtred Society</a> &middot; <a href="https://www.sliderulemuseum.com/">International Slide Rule Museum (ISRM)</a>
</div>
```

(See `chapter-4-specialty-rules.md`'s Model 4165 Urea Index Rule for the
full worked example.)

- **`**bold**` markdown does not process inside this raw HTML block** —
  use real `<strong>` if needed.
- Fields are `What it is / What it did / Approx. introduced` — this is
  the K&E-specific triad (distinct from `gallery-ai-stub`'s astronomy
  fields). Populate with whatever real facts were found (dimensions,
  material, scale purpose, catalog years), not generic filler.

## Distinct from Chapter 7's portfolio placeholder — don't conflate

`ke-stub-placeholder` is for rules Jay does **not** own/hasn't verified,
where facts (if any) come from AI/external research. A *different*
situation — a rule Jay **does** own but just hasn't written up yet —
uses a plain italic sentence ("I own this rule but haven't written up
its details yet — check back as this catalog grows") with **no stub
box at all**, since no facts are being asserted, AI-researched or
otherwise. See `project_ch7_portfolio_accordion` memory. Ownership-
without-a-writeup-yet ≠ unowned-with-AI-research — don't box the former.

## How to apply

1. Give the model its own heading regardless of how little material
   exists for it.
2. Check McCoy's catalog site and osgalleries.org for real photos/specs
   before assuming pure speculation is the only option.
3. Apply the authorship test above to decide stub-box vs. plain sourced
   prose.
4. Populate real facts into the fields wherever they exist; disclaim
   clearly; link real sources in the footer (McCoy's page if used, plus
   the standard Oughtred/ISRM links).
5. Self-audit before presenting a chapter: did any bare model get merged
   into another's section or bullet-listed instead of given its own
   heading?

## Verify

Build and spot-check the rendered box — badge, fact grid, disclaimer,
photo (if present) with working `data-hires` lightbox, and footer links
all render correctly; confirm every link target is a real, live URL
before hardcoding it.
